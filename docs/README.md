# docs 目录说明

> 这个项目有很多 .md 文件，每个职责不同。这篇讲清楚**哪个文档看什么、写什么**，别搞混。

---

## 一图看清

```
docs/
├── README.md           ← 你在看的这篇（导航）
├── architecture.md     ← 架构蓝图（项目长什么样、技术栈）
├── known-issues.md     ← 已知未解决问题（待办）
├── troubleshooting.md  ← 已修复的坑（经验库）
├── adr/                ← 重大决策记录（为什么选 A 不选 B）
├── notes/              ← 开发随笔（人话写的思考）
└── plan/               ← 阶段计划与验收（任务清单）
```

另外项目根目录有 `AGENTS.md`，也属于文档体系。

---

## 各文档职责

### `AGENTS.md`（项目根目录）

**是什么**：给 AI 看的工作规则。

**内容**：禁用 any、用箭头函数、软删规则、审计列必须 omit……都是**命令式**的规矩。

**什么时候看**：每次让 AI 写代码前，它都会读这个。

**写什么**：硬规则。发现 AI 反复犯某个错，就把规则写进来。

---

### `docs/architecture.md`

**是什么**：项目架构蓝图。

**内容**：技术栈、目录结构、核心约定（模块三件套、横切关注点 = plugin）、反例参考、借鉴清单。

**什么时候看**：想了解"这个项目整体长什么样"。

**⚠️ 注意**：蓝图是**理想态**，和实现会有偏差（见 notes《文档和实现会脱节》）。定期校对。

---

### `docs/plan/`

**是什么**：分阶段开发计划。

**内容**：7 个阶段（stage-1 到 stage-7），每个有子任务清单 + 验收清单 + 避雷。`README.md` 是进度看板。

**什么时候看**：开工某个阶段前读对应的 stage-N；想知道整体进度看 README.md。

**写什么**：完成任务后勾选验收项，附 commit 引用。

---

### `docs/adr/`（Architecture Decision Record）

**是什么**：重大架构决策记录。

**内容**：格式严格（背景/决策/理由/反对方案/后果）。只记**有显著取舍、长期影响**的决策，比如"认证表第一版不拆"。

**什么时候看**：想了解"为什么当初这么选"。

**⚠️ 克制**：不是所有决定都写 ADR。BizError 用 class 还是工厂这种小选择不写（见 notes 相关讨论）。ADR 是追加式的——推翻旧决策就写新的 ADR-0002，不改旧的。

---

### `docs/notes/`

**是什么**：开发随笔，**人话写的思考**。

**内容**：技术选型的真实理由、踩坑后的反思、对某个设计的顿悟。不学术、不结构化，像跟未来的自己聊天。

**什么时候看**：想回忆"我当初是怎么想的"。

**写什么**：让你有"哦原来如此"的时刻就记一条。不用每件事都记。README.md 里有详细说明。

**和其他文档的区别**：
- ADR 记"决定了什么"，notes 记"我想通了什么"
- troubleshooting 记"怎么修"，notes 记"为什么会这样"

---

### `docs/troubleshooting.md`

**是什么**：已修复的坑库。

**内容**：现象/原因/修复三段式。比如"refine 标注 z.ZodType 会击穿类型"。

**什么时候看**：遇到报错先来这里搜。

**写什么**：每次踩坑修完，记一条。重点是**可复现的修复方案**，不是思考过程（思考去 notes）。

---

### `docs/known-issues.md`

**是什么**：已知但未解决的问题。

**内容**：现象/根因/为何暂不修/何时回来修。比如"数字枚举 Scalar 渲染难看（zod v4 限制）"。

**什么时候看**：想知道"还有什么没解决"。

**写什么**：发现但暂时不修的问题。修完后从这里删掉（或移到 troubleshooting）。

---

## 一句话区分容易混的几个

| 纠结写哪 | 问自己 |
|---|---|
| ADR vs notes | **这是架构决策，还是个人思考？** 决策去 ADR，思考去 notes |
| notes vs troubleshooting | **这是"为什么"，还是"怎么修"？** 为什么去 notes，怎么修去 troubleshooting |
| troubleshooting vs known-issues | **这个坑修了吗？** 修了去 troubleshooting，没修去 known-issues |
| AGENTS.md vs architecture.md | **这是规则还是说明？** 规则（"必须 omit 审计列"）去 AGENTS.md，说明（"审计列是什么"）去 architecture.md |
| plan vs 其他 | **这是任务进度吗？** 是，去 plan；不是，去对应文档 |

## 文档之间的关系

```
AGENTS.md（规则）
    ↓ 指导
architecture.md（蓝图）──校对──→ 实际代码
    │
    ├─计划→ plan/（任务）
    │         └─完成→ 勾选 + commit 引用
    │
    ├─决策→ adr/（为什么这么选）
    │
    ├─思考→ notes/（我悟了什么）
    │
    └─踩坑→ troubleshooting（已修）/ known-issues（未修）
```

**核心原则**：文档和代码是两个会独立漂移的东西。定期校对，别让偏差累积成误导。
