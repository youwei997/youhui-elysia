# CRUD 做完不等于接口完成

`2026-06-23` · 阶段 4.2·4.3 对齐前端后的顿悟
---

## 起因

今天干了两件"低级"活：

- 给 user 补 `options`、`form`、`keywords`、`deptId`、密码重置、批量删除
- 给 role 补 `options`、`form`、`menu-ids`、`dept-ids`

看 commit 记录会觉得："啊？阶段 4.2 不是做完了吗？怎么今天都在补？"

## 我之前以为的

三件套（schema / queries / routes）走完，一个模块就交付了。CREATE / UPDATE / FIND / DELETE 齐活，Swagger 文档整整齐齐，任务就该标 ✅。

## 真做的时候发现不这样

前端需要的端点，标准 REST 没教过：

```
GET  /users/:id/form        ← 编辑表单预填
GET  /users/options         ← 下拉框要 [{ value, label }]
GET  /roles/:id/menu-ids    ← 树选择器要 Flat Array [1,2,3]
GET  /menus/routes          ← 把平面菜单转成嵌套路由树
DELETE /roles/1,2,3         ← Element UI 表格点批量删除就这么发
```

更隐蔽的是形参——前端表格要 `keywords` 模糊搜、`deptId` 过滤，标准分页接口不声明这些字段。

这些端点**不在任何 REST 教程里**，但它们真实存在，因为 UI 控件就是这么调后端的。

## 那谁错了

都没错。问题在于：

**后端 CRUD 描述的是数据模型的完整生命周期，前端接口描述的是 UI 控件的调用约定。** 两者不是一一映射的关系。

- `/users` 列表 → 表格组件调
- `/users/options` → Select 组件调
- `/users/:id/form` → Form 组件调
- `/users/:id` → 详情页调

同一个数据，四套 UI 就要四套端点（至少）。这跟"重复封装"不是一回事——每个端点的数据形状和权限语义都可能不同。

## 想通后怎么看"填坑期"

补 `options`、补 `form`、补批量删除、补树形路由……**这段工作不是打补丁，是后端对齐前端交互建模完成度的度量。**

做完三件套只代表"数据生命周期都通了"，不代表"前端每个控件都能跑"。相当于后端把毛坯房建好了，但水电插座开关都还没装——房子不能住人，跟没盖一样。

## 一个判断 checklist

下次做完模块三件套，先别急着标完成，拿这套问题问前端：

1. 下拉框要调哪个端点？（`options`）
2. 编辑表单需要预填哪些数据？是单表还是关联拼装？（`form`）
3. 有没有批量操作？端点路径长什么样？
4. 搜索框除了分页 param，还有没有额外过滤条件？（`keywords`、`deptId`）
5. 树形选择器要的是平面数组还是嵌套结构？

前五个问题清完，这个模块才算真的"前端可用"。

## 一个小感悟

写后端容易陷入"接口设计的自我感动"——追求 REST 优雅、资源命名统一。真到了对接阶段发现，前端的接口清单里总有一批"看起来不该存在"的路由。这些不是前端乱调，是 UI 控件的客观需求被标准 CRUD 漏掉了。

**模块完成度 = 三件套 + 补齐前端的隐性端点清单。** 少一条，交付就是假的。
