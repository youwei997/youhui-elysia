# 有现成前端时，API 怎么定不看你

`2026-06-23` · 阶段 4.2→4.3 对齐前端 API 后的反思

---

## 起因

最近在做 RBAC 模块，对齐 vue3-element-admin 的 API 时，创建了一堆"看起来不像标准 REST"的端点：

```
GET  /users/options        → [{ value, label }]   for 下拉框
GET  /users/:id/form       → { ...user, roleIds }  for 编辑表单
GET  /roles/options        → 同上
GET  /roles/:id/form       → { ...role, deptIds }  for 编辑表单
GET  /roles/:id/menu-ids   → [1, 2, 3]             for 菜单树选择器
GET  /roles/:id/dept-ids   → [4, 5, 6]             for 部门选择器
GET  /menus/routes         → 把平面菜单转成 RouteItem[] 嵌套结构
```

最狠的是 `DELETE /api/v1/roles/1,2,3`——URL 路径里传逗号分隔的 IDs，REST 洁癖看了会吐血。

写的时候没想太多，写完回头看，发现所有"奇怪"端点都有一个共同特征：**前端已经这么调了，我只是照抄。**

## 之前以为

后端应该先设计一套"优雅的 REST API"——HTTP 方法对、资源路径清晰、状态码规范——然后前端来适配我。API 规范的标准就是 REST 教条。

按这个思路，「正确的」API 应该是：

```
GET    /roles/:id          # 一个端点覆盖所有读场景
DELETE /roles               # body: { ids: [1,2,3] }
```

前端需要的"编辑时的角色选项"自己从 `/roles` 列表拼，需要的"菜单 ID 列表"自己从 `/roles/:id` 的返回体里取 `menus[].id`。

## 真做的时候发现不对

**1. 改前端的成本远大于改后端**

`GET /roles/:id/form` 在后端就是两行查询拼一起：`findRoleById(id)` + `findRoleDeptIds(id)`。如果改前端，需要改组件的数据解构、TypeScript 类型定义、表单联动逻辑。一个端点改了，影响面是前端 3-4 个文件，每个都可能引入 bug。

**2. 前端和服务端"看到的世界"不一样**

下拉选择器要的是 `[{ value, label }]`，不是完整的用户对象。表格列表要的是分页数据，不是树形结构。菜单树要的是嵌套 children，不是平面数组。不是"前端懒"，是每个 UI 控件看到的确实就是不同的数据形态。

**3. 谁叫谁改**

前端是用户看到的，后端是用户看不到的。后端多做 1 个 endpoint 加 10 行代码，用户没感觉。改前端，用户会说"上次还好好的，这次怎么坏了"。

**4. DELETE 用逗号分隔 ID**

Element UI 的表格批量操作就是发 `DELETE /api/v1/roles/1,2,3`。前端不是我写的，是开源的。我有什么资格让它改？

## 想通后的判断

**有现成前端时，API 契约的"正确性"标准是「前端不改」，不是「RESTful 规范」。**

这套前端不是我写的——它是一套已经跑通的系统。我的后端是后来者。后来者适应已有系统，不是反过来。

「优雅」是给新项目看的，但现实里的项目接口都是和前端商量着长的。没有前端黏土，你要什么形状捏什么形状。有前端黏土，形状已经定了，后端是灌进去的石膏。

REST 是指导原则，不是交规。不写在《中华人民共和国 REST 法》里的，都可以让。

## 口诀

**API 的规范不是你看 REST 教程学的，是前端已经调好了你再原样实现的。**
