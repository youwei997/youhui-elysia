# Dept 模块设计要点

> 阶段 4.4，部门管理模块——树形结构的 CRUD 到底难在哪

---

## 1. 树形结构：部门管理的核心复杂度

部门跟用户、角色最大的区别是：**部门有层级**。

- 用户是平铺的列表，查就完了
- 部门是一棵树，涉及"父部门-子部门"的关系

一棵树的常见操作：

```
新增：选一个父部门，挂到它下面
编辑：可能换父部门（比如从研发部移到财务部）
删除：删掉一个部门，它下面的子部门怎么办？
查询：拿到的是一棵完整的树
```

这棵树长什么样：

```
公司总部
├── 研发部
│   ├── 前端组
│   └── 后端组
└── 财务部
```

## 2. 树是怎么存到数据库的？

数据库是"表"，不是"树"。怎么把树塞进表里？

最简单的方式：**每条记录记一个 parentId**。

| id | name | parentId |
|---|---|---|
| 1 | 公司总部 | 0 |
| 3 | 研发部 | 1 |
| 5 | 前端组 | 3 |

`parentId=0` 表示根节点。查的时候：查出所有记录，在内存里组装成树。

这就是我们这个项目的做法。

## 3. 但是有个坑：子树查询

需求来了：**数据权限**——某个角色只能看"研发部及其子部门"的数据。

如果只有 parentId，你要查"研发部(3)及它下面的所有子部门"，你怎么查？

- 方案 A：先查出 3，再查 parentId=3 的（前端组），再查 parentId=前端组的……一层层递归
- 方案 B：每条记录额外存一个字段，记录"从根到我的路径"

我们选了方案 B，这个字段叫 **treePath**。

## 4. treePath 是什么

就是一条"路径记录"，像文件路径一样：

| id | name | parentId | treePath |
|---|---|---|---|
| 1 | 公司总部 | 0 | "0" |
| 3 | 研发部 | 1 | "0,1,3" |
| 5 | 前端组 | 3 | "0,1,3,5" |

规则很简单：**父节点的 treePath + "," + 自己的 id**。

- "0" 是虚拟根（所有根节点的父都是 0）
- 看 treePath 就知道它上面有哪些祖先：前端组的 "0,1,3,5" → 一眼看出上面有公司总部(1)和研发部(3)
- 查子树也简单：查 treePath **以 "0,1,3" 开头**的记录，就能拿到研发部及它下面的所有部门

这就是 treePath 的核心价值：**一次查询就能拿到整棵子树，不需要递归**。

## 5. treePath 的代价：改父部门时要修路

treePath 好查，但不好改。

假设前端组从研发部移到财务部下面：

```
改之前：
  前端组：treePath = "0,1,3,5"   ← 前缀是 "0,1,3"
  前端组-前端小组：treePath = "0,1,3,5,8"  ← 前缀也是 "0,1,3"

改了之后：
  前端组：treePath = "0,1,4,5"   ← 新前缀 "0,1,4"
  前端组-前端小组：？？？        ← 还留在 "0,1,3,5,8"，脏了！
```

所以**改父部门时，要把所有子节点的 treePath 前缀一起换掉**。这个操作必须在同一个事务里完成——要么全改对，要么不改。

好在"改父部门"这个操作极少发生，代价可以接受。

**取舍总结：用"更新时的麻烦"换"查询时的快"——因为部门树读多写少。**

## 6. 防循环：一个容易忽略的问题

假设：

```
公司总部 → 研发部 → 前端组
```

如果把公司总部的父部门改成前端组，就变成：

```
前端组 → 公司总部 → 研发部 → 前端组 → ...
```

**循环了，永远找不到根**。

所以改父部门时必须检查：**新父部门不能是目标节点自己，也不能是目标节点的子孙**。

怎么查？很简单：看新父节点的 treePath 里有没有目标节点的 id。如果有，说明目标节点是新父的祖先，那就形成循环了。

## 7. 删除部门：级联操作

删研发部的时候，前端组和后端组怎么办？

有三种选择：

1. **禁止删除**：只要部门下有子部门或用户，就不让删
2. **子部门升一级**：子部门变成父部门的同级
3. **级联删除**：父删了，孩子一起删

我们选了 **3 + 1 的组合**：
- 子部门：**级联删除**（一起软删）
- 用户：**禁止删除**（部门下有人就不让删）

为什么？部门是组织架构，删了父部门子部门没有存在的意义。但用户不能跟着一起删，用户属于公司不属于部门。

## 8. 三条线：职责分工

```
前端请求 → routes（校验 + 编排） → queries（纯数据操作） → 数据库
```

- **routes**：检查参数对不对、有没有权限、有没有循环、父存不存在
- **queries**：只做数据库操作（查、增、改、删），不关心 HTTP 错误

routes 是"管家"，queries 是"干活的"——queries 不用知道"404 是什么"，它只管返回数据或 undefined。

## 9. 前端视角：树形 CRUD 的完整流程

### 查（页面加载）
```
请求 GET /depts/tree
  → 后端查出所有部门（扁平列表）
  → 内存里组装成树
  → 返回嵌套 JSON
  → 前端绑定到 el-tree / Tree 组件
```

### 增（新增部门）
```
打开 Dialog
  → 选上级部门（用 TreeSelect 组件选父部门）
  → 填名称、编码、排序
  → 提交 POST /depts
  → 成功 → 刷新树
```

### 改（编辑部门）
```
打开 Dialog，回显数据
  → 如果要换上级部门：
     注意：不能选"自己"或"自己的子部门"作为上级
  → 提交 PUT /depts/:id
  → 成功 → 刷新树
```

### 删（删除部门）
```
弹确认框
  → 提交 DELETE /depts/:id
  → 如果部门下有用户 → 后端返回 400 "部门下存在用户" → 前端弹提示
  → 成功 → 刷新树
```

## 10. 跟 Menu 模块的对比

Dept 和 Menu 都是树，但差异点：

| 维度 | Dept | Menu |
|---|---|---|
| 数据权限 | 被其他模块引用（数据权限要用到部门树） | 只做权限校验 |
| 特有字段 | 编码、部门负责人 | 类型(目录/菜单/按钮)、权限标识、路由路径、图标 |
| 特有端点 | 无 | `/menus/routes`（前端动态路由） |

---

**一句话总结 Dept 模块**：部门就是一棵树，核心就三件事——**查子树用 treePath 一次搞定，改父部门要修所有孩子的路，删部门要带孩子一起走但别动用户**。
