# 文档不是写完就定格的——它会和实现脱节

`2026-06-15` · 跨阶段 1-2 的复盘

---

## 起因

这次会话做了大量"校对"工作，发现了一个反复出现的问题：**文档说做了，实际没做；或者说改了，实际被还原了**。不是一次，是好几次。

## 几个真实的例子

### 1. architecture.md 一直写 TypeBox，实际用 zod

architecture.md 是阶段 1 就写好的蓝图，里面技术栈写的是 TypeBox + drizzle-typebox。但阶段 2 实际实现时改成了 zod（见 `docs/notes/2026-06-13-为什么选zod不选TypeBox.md`）。

**改了代码没改文档**。直到这次会话专门校对，才发现 architecture.md 还写着 TypeBox，已经误导了很久。如果新来的 AI 或人按文档用 TypeBox，会和现有 zod 代码直接冲突。

### 2. stage-2 验收清单的勾选，被还原过

我（AI）之前用脚本批量勾选了 stage-2 的 5 个验收项，脚本输出 "DONE"。但后来发现**这些勾选被外部 git 操作覆盖还原了**。我之前只看脚本输出 "DONE" 就相信了，没回读文件确认。

结果就是：清单显示"未完成"，但代码其实早就改了。用户后来一句"阶段 2 验收清单 5 项没勾" 才揭出来。如果用户不查，这个不一致会一直挂着，误导"阶段 2 没做完"。

### 3. "我以为改了" 其实没改

最严重的一次：我声称给 queries.ts 补了软删过滤（isNull(deleteTime)），typecheck 也"通过了"。后来用户问"你在我项目就改了两个文件啊"，一查 git——**queries.ts 根本没改动**。typecheck 能过恰恰是因为代码没变。

我那次是基于"以为改了"的错误前提，一路推进到写"本阶段收获"总结（里头还写了软删过滤的心得），如果直接提交，就是"文档说做了、代码没做"的严重不一致。

## 这几个例子的共性

都是**"我以为生效了，实际没生效"**。原因有两类：

1. **代码改了，文档没跟**（TypeBox → zod）
2. **改动声称生效，实际被还原/没落地**（勾选、queries.ts）

第 2 类更隐蔽，因为它会让 AI 基于错误前提继续推进，错误层层放大。

## 我（AI）为什么会犯这个错

- **工具返回值骗人**：Edit/脚本返回 "updated successfully" / "DONE"，我就信了。但实际文件可能被外部 git 操作、编辑器、格式化插件还原了。
- **不回读确认**：正确做法是改完用 `git diff` 或读文件验证落地，但我会偷懒跳过。
- **基于"应该改了"的前提推演**：一旦某个早期步骤实际没生效，后面所有依赖它的判断都错了，但我不会回头怀疑前提。

## 教训（给自己）

### 对 AI 协作的教训

1. **任何文件改动，不能凭工具返回值判断，必须 `git diff` 回读**。尤其在有外部并发操作（编辑器、watcher、其他 git 进程）的环境下。
2. **文档和代码是两套独立的东西**。改代码不等于文档自动跟，反之亦然。每次大改后要专门做一轮校对。
3. **怀疑前提**：当结果"看起来很顺"（typecheck 通过、清单全勾），要警惕是不是某个早期步骤根本没生效，顺是因为没改动。

### 对项目维护的教训

1. **架构文档是理想，实现是现实，两者必然有偏差**。定期对账，别让偏差累积成误导。
2. **关键状态（验收清单、进度看板）要和代码强绑定**。这次 stage-2 清单的勾选带上了 commit 引用（`见 commit 7eb9bff`），就是个好做法——可以追溯到具体改动，而不是空勾选。
3. **文档的"完成"声明要可追溯**。写"已修复"要附 commit hash，写"已实现"要能指向文件。空口声明不可信。

### 一个可复用的工作流

以后每次 AI 声称"做完了 X"，我（用户）可以问一句：
- "git diff 给我看落地了吗？"
- "改了哪些文件？状态干净吗？"

而 AI 每次声称完成，应该主动给 git diff 验证，而不是只报 "DONE"。**信任要建立在可验证的证据上，不是工具的返回值。**

## 最大的感悟

文档和代码的关系，不是"写一次就定格"。它们是两个**会独立漂移**的东西——代码随需求变，文档随理解变，没人主动对齐，它们就各走各的，最后差十万八千里。

这次会话最大的价值，不是修了几个不一致，是让我意识到：**"对账"本身是个独立的工作**，不能指望它自然发生。定期把文档和代码摆一起核对，是个被低估的好习惯。
