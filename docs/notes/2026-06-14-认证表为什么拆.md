# 认证表为什么大家都拆两张

`2026-06-14` · 阶段 3 开工前

---

今天讨论阶段 3 要不要拆认证表，我随口问了句"你为什么不按主流命名"，结果挖出来一个挺有意思的事。

## 起因

文档里原来管它叫 `auth_credential`。我说好像在 Convex 见过类似的，问 AI 为什么不按主流命名。AI 一查才发现，**几乎所有现代认证库都拆两张表**，而且名字都不叫 credential：

- Convex Auth → `users` + `authIdentities`
- Supabase → `auth.users` + `auth.identities`
- NextAuth → `user` + `account`
- Better Auth → `user` + `account`
- Lucia → `user` + `key`

我当时就"哦"了一下——原来这不是某个库的怪癖，是**行业共识**。

## 为什么都拆

想了下其实很自然。一旦支持 Google 登录 + 微信登录 + 密码登录，一个用户就有**好几个登录身份**。塞 users 表一个 password 字段根本装不下，必须另开一张表，每个登录方式占一行。

还有个我没想到的好处：**登录时只扫小表**。不用把整个用户档案（头像、昵称、部门...）都拉出来，就查个"这个账号对不对"就行，快。

## credential 这个词为啥不好

AI 解释我才反应过来。credential 听着像"密码/凭证"，但 Google OAuth 登录时我们存的根本不是密码，是 Google 给的一个标识（provider + subject）。那个东西**叫身份（identity）或账户（account），不叫 credential**。

所以 `auth_credential` 这名字其实是个**语义错位**——它只适合描述密码登录，不适合描述 OAuth。文档里那个命名是错的，AI 当时直接照搬没质疑，我也没多想就接受了。

## 一个小感悟

这件事让我意识到：**AI 不会主动质疑文档/需求**。它拿到 `auth_credential` 这个词，就直接用了，不会去想"这个词对不对"。是我多嘴问了一句"为什么不按主流"，才把这个错挖出来。

所以以后看 AI 的产出，不能光看"能不能跑"，还得想想"**这个选择本身合理吗**"。AI 给的东西默认是"够用"，不一定是"对"。

## 最后的决定

第一版还是不拆（阶段 3 只做密码登录，拆了是过度设计）。但记下来了：
- 未来要拆的话，叫 `auth_accounts`，不要叫 credential
- 详细理由写在 `docs/adr/0001-auth-table-not-split.md`

这条笔记是**认知**（我搞懂了为什么拆），ADR 是**决策**（我们决定暂时不拆）。两码事。
