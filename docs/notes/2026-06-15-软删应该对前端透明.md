# 软删应该对前端透明——以及防护为什么不能松

`2026-06-15` · 阶段 2 收尾

---

## 我是怎么一步步想清楚的

### 第一层：被漏洞驱动

最早注意到 deleteTime 不对劲，是因为做 Create/Update Body 收紧时发现一个反软删漏洞——前端只要在 PUT body 里塞 `"deleteTime": null`，就能把已删记录"复活"。于是给 Body 加了 `.omit({ deleteTime: true })`。

这个动作的**直接动机是堵漏洞**，还没想到"要不要在 Response 里也排除"。

### 第二层：经验触发的回想

漏洞修完之后，回想起自己以前做过的 PigX 项目——那个后台管理框架的响应里，**从来没有 deleteTime 或 isDelete 这种字段**，只有 createdAt / updatedAt 这种时间审计。

再去对比若依、JeecgBoot，发现全是这样：

| 框架 | 软删字段 | 前端能看到吗 |
|---|---|---|
| 若依 RuoYi | `del_flag` | ❌ |
| PigX | `del_flag` | ❌ |
| JeecgBoot | `del_flag` | ❌ |

这才反应过来：**不止是漏洞要堵，软删标志本身就不该出现在响应里**。它对前端语义无意义（"这条数据 3 天前被删过"对前端没价值），还会暴露内部状态污染 API 契约。

于是把 `UserResponse` 也加了 `.omit({ deleteTime: true, createdBy: true, updatedBy: true })`，保留 id 和 createdAt/updatedAt。

### 第三层：那防护还需要吗？

Response 也排除掉 deleteTime 之后，我突然想：**如果前端始终拿不到 deleteTime，那 Create/Update Body 里的"软删防护"还需要做吗？**

想了想，**需要，而且不能去掉**。

#### Response 排除 ≠ 请求无法到达

关键认知：前端 UI 没这个字段，不代表**有人不能传**。API 是公开的 HTTP 端点：

```bash
# 前端 UI 不会发这个，但攻击者可以直接发
curl -X PUT localhost:3000/users/3 \
  -d '{"deleteTime": null}' \
  -H "Content-Type: application/json"
```

zod schema 的 omit 是**请求体校验**——它在进 handler 之前就拒绝 deleteTime。如果去掉，`db.update().set(data)` 会把这个 null 写进数据库。

#### 两道防线缺一不可

- **Response omit**：对正常用户友好（UI 干净，不困惑）
- **Body omit**：对攻击者防御（恶意请求被 zod 拦）

这是**纵深防御（defense in depth）**。两道防线针对不同对象，删任何一道都有缺口。

同理：Response 里有 id（前端要用），但 Create/Update Body omit 了 id——因为 id 应该从 URL 路径来，不该从 body 来。**请求和响应是两个独立契约，不能因为一个排除了就放松另一个。**

## 主流框架的做法：双层隔离

若依 / PigX 的"透明"其实是**两层隔离**叠加的效果：

1. **查询层自动过滤**——MyBatis-Plus 的 `@TableLogic`，所有 select 自动加 `where del_flag = 0`，业务代码不用手动写
2. **响应层不暴露**——返回给前端的 DTO 单独定义，不含 del_flag

前端从头到尾不知道有"软删"这回事。它只知道"查不到了"（因为被过滤了）。

我们这个项目对应：
- 查询过滤：手动 `isNull(deleteTime)`（Drizzle 没有全局过滤器）
- 响应不暴露：`UserResponse.omit({ deleteTime: true })`
- 请求拒绝：`UserCreateBody/UserUpdateBody.omit({ deleteTime: true })`

比主流框架多了一道（请求拒绝），因为 Drizzle 派生的 DTO 默认全暴露，不主动 omit 就会漏。

## 感悟

### 思考是分层的，别把后一层当全部原因

如果有人问我"为什么 Response 要排除 deleteTime"，最简短的答案是"漏洞驱动"。但真实过程是三层：漏洞驱动 → 经验验证 → 纵深防御反思。如果只记第一层，会忽略"Response omit 和 Body omit 是两道独立防线"这个关键认知。

以后复盘一个决策，多问自己："**这个决策是只有一个原因，还是好几层原因叠起来的？**"

### "我以前见过的不一样"是值钱的直觉

这次的核心触发点，是 PigX 的经验记忆。AI 不会主动告诉你"主流框架不暴露 deleteTime"——它只会按当前文档照做。但经验会本能地提醒"这不对劲"。

所以以后做新东西，多想想"我以前见过的同类东西长什么样"。**经验对比能挖出 AI 照搬文档时漏掉的常识。**

### Drizzle 的局限是纪律的代价

MyBatis-Plus 的 `@TableLogic` 让软删过滤对开发者无感。Drizzle 没有全局过滤器，只能每个查询手动加、每个 DTO 手动 omit。漏了就是 bug（我们确实漏过）。

这是 AGENTS.md 里"软删过滤规则表"和"审计列必须 omit"存在的根本原因——**只能靠纪律弥补工具的不足**。工具没有的能力，纪律补上；工具有的能力（比如 MyBatis 的 @TableLogic），纪律就可以简化。
