# 软删应该对前端透明

`2026-06-15` · 阶段 2 收尾

---

## 起因

我在 `UserResponse` 里发现 `deletedAt` 被返回给前端了。当时就觉得不对——以前做过一个 PigX 微服务的前端项目，那个后台管理框架的响应里，**从来没有 deletedAt 或 isDelete 这种字段**，只有时间审计和业务字段。

于是问 AI：主流框架（若依、PigX）到底怎么处理软删的？

## 主流框架的做法

AI 查了一圈，发现若依、PigX、JeecgBoot 这些后台框架，软删对前端**完全透明**：

| 框架 | 软删字段 | 前端能看到吗 |
|---|---|---|
| 若依 RuoYi | `del_flag` | ❌ 看不到 |
| PigX | `del_flag` | ❌ 看不到 |
| JeecgBoot | `del_flag` | ❌ 看不到 |

它们靠**两层隔离**：

1. **查询层自动过滤**——MyBatis-Plus 的 `@TableLogic`，所有 select 自动加 `where del_flag = 0`，业务代码不用手动写
2. **响应层不暴露**——返回给前端的 DTO 单独定义，**不含 del_flag**

前端从头到尾不知道有"软删"这回事。它只知道"查不到了"（因为被过滤了），不会困惑"为什么这条数据有 deletedAt"。

## 为什么不该暴露

想了一下，理由挺直接的：

1. **语义无意义**——对前端而言，数据要么存在要么不存在。"这条 3 天前被删过"对前端没任何价值，反而困惑
2. **安全风险**——前端要是能塞 `"deletedAt": null`，就能把已删记录"复活"（反软删）。这个坑我们之前修 Create/Update Body 时踩过，Response 这边居然还漏着
3. **API 契约干净**——前端依赖"业务字段 + 审计时间"，混入"内部状态字段"污染契约

## 这个项目的实际情况

发现 `UserResponse` 只 omit 了 `password`，审计列 + deletedAt 全暴露：

```ts
// 之前
export const UserResponse = createSelectSchema(sysUser)
  .omit({ password: true })  // ← 只排了密码！
```

改成了：

```ts
export const UserResponse = createSelectSchema(sysUser)
  .omit({
    password: true,
    deletedAt: true,      // 软删标志，前端不该看到
    createdBy: true,       // 内部 id 无意义
    updatedBy: true,       // 阶段 4 会 join 成创建人名字
  })
```

保留了 `id`（前端要用）、`createdAt` / `updatedAt`（前端列表显示时间列，和若依/PigX 一致）。

## 一个小感悟

这次是从**自己以前做过的项目经验**里挖出来的——PigX 前端没见过 del_flag，所以下意识觉得"暴露 deletedAt 不对"。

这种"我以前见过的不一样"的直觉特别值钱。它不是理论推导出来的，是**经验沉淀的成本能**。AI 不会主动告诉你"主流框架不暴露 deletedAt"——它只会按文档照做。但如果你的经验告诉你不对劲，就要追问。

所以以后做新东西，多想想"**我以前见过的同类东西长什么样**"。这个对比往往能挖出 AI 照搬文档时漏掉的常识。

## Drizzle 的局限

另外发现一个 Drizzle 和 MyBatis-Plus 的差距：**Drizzle 没有全局查询过滤器**。

MyBatis-Plus 的 `@TableLogic` 让所有 select 自动加 `del_flag = 0`，业务代码不用管。Drizzle 要每个查询手动加 `isNull(deletedAt)`，漏了就是 bug（我们之前就漏过）。

这是 AGENTS.md 里"软删过滤规则表"存在的根本原因——**只能靠纪律弥补工具的不足**。Drizzle 未来如果加全局过滤器，这个规则就可以简化，但现在得认。
