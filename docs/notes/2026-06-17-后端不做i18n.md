# 后端不做 i18n

## 决策

阶段 3.7 i18n plugin **跳过**，不实施。

## 调研结论

参考项目 youlai-boot（Spring Boot 后端）和 vue3-element-admin（Vue3 前端），**两端都没做"错误码 → 多语言"映射**：

### 后端 youlai-boot

`ResultCode.java`（错误码枚举）里**直接写死中文**：

```java
ACCOUNT_NOT_FOUND("A0201", "用户账户不存在"),
ACCOUNT_FROZEN("A0202", "用户账户被冻结"),
USER_PASSWORD_ERROR("A0210", "用户名或密码错误"),
```

`src/main/resources` 下面**没有** `i18n/` 目录，没有 `messages_zh.properties` / `messages_en.properties`，**没有任何 Locale / Accept-Language 引用**。

### 前端 vue3-element-admin

`src/lang/package/zh-cn.json` 和 `en.json` 里**只有 UI 文案**（按钮、菜单、表单提示），**没有错误码字典**。

`src/utils/request.ts` 错误处理**直接显示后端返的 msg**：

```ts
ElMessage.error(msg || "系统出错");
```

不翻译，不做 code → 文案映射，**后端返什么就显示什么**。

## 为什么不做后端 i18n

1. **参照项目没做**——你抄的是 youlai，youlai 不做就不做。强行做就是造差异
2. **前端已经够用**——vue-i18n 翻 UI 文案（按钮/菜单/表单）就够覆盖 90% 用户场景
3. **后端 i18n 是无底洞**——加语言加错误码改文案都得动后端，且要做 Accept-Language 解析、q 值排序、locale fallback
4. **后端业务错误消息基本就是字典**——`A0201` → "用户账户不存在"，前端拿到 code 用 vue-i18n 翻译完全没问题
5. **AGENTS.md 第 14 行**——"第一版只追求'对'，不追求'快'"

## 错误消息的分工

| 端 | 职责 | 实现 |
|---|---|---|
| 后端 | 返错误码 code + 中文 msg（来自 ERR_MSG 字典） | lib/errors.ts |
| 前端 | UI 文案中英切换（按钮、菜单、表单） | vue-i18n |
| 前端 | 业务错误消息**直接显示后端返的 msg** | ElMessage.error(msg) |

## 未来如果真要做国际化

最简单的方案：把 `lib/errors.ts` 的 `ERR_MSG` 拆成字典对象，按 locale 选：

```ts
const ERR_MSG = {
  'zh-CN': { A0201: '用户账户不存在', ... },
  'en-US': { A0201: 'Account not found', ... },
}

const t = (code, locale) => ERR_MSG[locale]?.[code] ?? ERR_MSG['zh-CN'][code]
```

不用做 plugin，不用解析 Accept-Language，半小时的事。**但现在不做**——理由同上 1-5。

## 我之前的误解

以为"国际化"是后台管理框架的标配。看了 youlai 才发现：后台管理框架的 i18n **都是只翻 UI 文案**，业务错误消息直接用后端语言。

这个误解是**把"应用 i18n"和"框架 i18n"混为一谈**。前者是产品决策，后者是工程决策。你这个阶段是搭框架，不是做产品，所以框架层不做 i18n 是合理选择。
