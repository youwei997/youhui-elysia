# 为什么 token 注销用黑名单不用白名单

## 背景

设计 token 失效机制时有两种思路：

- 白名单：登录时把 token 写进 Redis，每次请求查 Redis，有 = 有效
- 黑名单：登出/踢人时把 jti 写进 Redis，每次请求查 Redis，有 = 已注销

我之前一直以为黑名单只比白名单实现复杂一点，实现完发现其实有更深的取舍。

## 之前以为的黑名单优点

- 正常路径不写 Redis，登录快
- 存储量小

听起来都挺弱的，直到我看到 Redis 挂了 的场景。

## Redis 挂了的真实情况

白名单：

```
登录 -> 写 Redis
请求 -> 查 Redis（有才放行）
Redis 挂了 -> 全站 401
```

意味着 Redis 宕机 1 分钟，全站用户全部登出。这比密码泄漏还可怕。

黑名单：

```
登录 -> 不碰 Redis，只签 JWT
请求 -> 验签 + 查黑名单（查不到就放行）
Redis 挂了 -> 当成未注销放行
```

Redis 挂了，正常用户不受影响。已注销的 token 在 Redis 恢复前短暂恢复有效，这是个可接受的小窗口。

## 关键反转

我之前的逻辑是错的——

> 黑名单比白名单麻烦一点点
> 白名单把 Redis 当成唯一登录状态源，黑名单把 Redis 当成辅助

黑名单的真正优势是可用性，不是存储量。

## 取舍

- 接受：已注销 token 在 Redis 故障窗口内可能短暂复活
- 换来：登录不依赖 Redis、Redis 挂了正常用户能用、存储量小

## 实现细节

黑名单 key 的 TTL 设为 token 剩余有效期（PX remainingMs）：

- token 还活着，黑名单记录就还活着
- token 死了（exp 到了），黑名单自动消失

不设 TTL 的话，几十万用户登出会堆几十万条永远不删的记录。这个 TTL 是黑名单能 work 的关键。
