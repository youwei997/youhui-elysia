# traceId 为什么不进响应体

`2026-06-16` · 阶段 3 · 校对验收清单

---

验收清单里有一条 "响应包含 traceId"，看了一眼代码发现根本没实现。讨论了一下决定：**不补了，traceId 只留在日志侧。**

## 为什么

### 1. traceId 的消费者是后端开发，不是前端

traceId 的使用链路是：前端报错 → 后端 `grep` 日志 → 定位问题。这个链路从头到尾是后端内部的事，traceId 只是一个"日志索引键"。前端拿到它唯一的意义就是"帮后端转述"。

那不如让错误发生时就打全日志，后端直接去日志平台搜时间窗口，不需要前端传什么编号。

### 2. 多了个字段就多了份契约负担

响应体承诺 `{ code, msg, data }`，很干净。加了 traceId 后：
- 每个响应多了几个字节（虽然小，但量大了还是有成本）
- 前端类型定义要多一个字段
- OpenAPI 文档要多一个字段声明
- 如果有人依赖这个字段做链路追踪（比如前端打点上报），它就成了 API 契约的一部分，不能随便改

三个字段能说清楚的事，没必要加第四个。

### 3. 真的有需要再加，不迟

traceId 是 derive 注入的，error-handler 的 `onError` 回调里就能拿到 `reqId`。如果以后真有需要（比如对接 APM 平台、前端主动上报故障），在 error-handler 里加一行 `traceId: reqId` 的事。**不提前加，不提前绑。**

## 和前几篇笔记的关联

和《文档和实现会脱节》那篇说的东西是同一个模式：**文档写了，实际没做。** 但这次的区别是，讨论后决定"不做才是对的"，所以不是补代码，而是改文档预期。

和《为什么不用Result容器》也有点像——都是在问"这个看起来有用的东西，真放进响应里会怎样"，然后发现不加更干净。
