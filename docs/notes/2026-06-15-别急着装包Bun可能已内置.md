# 别急着装包——Bun 可能已经内置了

`2026-06-15` · 阶段 3 开工前

---

## 起因

准备阶段 3,我（AI）列了个依赖清单要装：

- `jose`（JWT 签名）
- `argon2` 或 `bcryptjs`（密码哈希）
- `ioredis`（Redis 客户端）

然后被用户连续三次纠正：

1. "Bun 不是内置 redis 驱动吗？"
2. "argon2 和 bcryptjs 怎么选？"（潜台词：为什么要装？）
3. "Elysia 不是推荐用 `@elysia/jwt` 生成 jwt 吗？"（给了官方文档链接）

三次都是同一个问题：**我照搬 stage-3 文档，没去核实 Bun/Elysia 的实际能力。**

## 真相：三个依赖只用装一个

| 功能 | 文档说装 | 实际 | 发现来源 |
|---|---|---|---|
| JWT | `jose` | `@elysia/jwt`（Elysia 官方插件，和 ctx 集成更好） | 用户给了官方文档链接 |
| 密码哈希 | `argon2`/`bcryptjs` | `Bun.password`（内置，默认 Argon2id，零依赖零 binding 问题） | 用户问"怎么选" |
| Redis | `ioredis` | `Bun.redis`（v1.2.9+ 内置，Zig 写的，比 ioredis 快 2-8 倍） | 用户问"是不是内置" |

**装包清单从 3 个 → 1 个**（`@elysia/jwt`）。Bun 三个内置能力里，两个完全不用装，一个换成官方插件。

## 为什么会犯错

stage-3 文档是阶段 1 写的蓝图。那时候可能：
- `Bun.redis` 还没出（v1.2.9 才加）
- `Bun.password` 已有但文档没提
- `@elysia/jwt` 被忽略了

文档写 `jose` + `ioredis` + `argon2` 在当时可能合理，但 **Bun 进化了，文档没跟**。这又是《文档和实现会脱节》那条笔记的活例子。

而我作为 AI 的失败模式：**拿到文档就把里头的"装 X 库"当成既定事实，没问"Bun 是不是已经有了"**。用户每次一个简单的问题就戳破——这种"反向校验"特别值钱。

## Bun 内置能力的认知更新

之前我对 Bun 的印象停留在"快速 JS runtime + 内置 TS + Bun.serve"。这次发现 Bun 已经内置了一堆"后端常用"的能力：

- `Bun.password` —— 密码哈希（argon2id / bcrypt，自动跑 worker 线程不阻塞）
- `Bun.redis` —— Redis 客户端（v1.2.9+，Zig 写的，RESP3）
- `Bun.sql` —— PostgreSQL 客户端（项目已经间接在用，通过 postgres.js，但其实 Bun 自己也有）

**教训：在 Bun 项目里，写"装 X 库"之前，先查 Bun 是不是已经内置了。** 内置的不用装，省依赖、省 binding 坑、性能还更好。

## 一个要留意的坑

查证过程发现 `Bun.redis` 有个**未解决的 reconnect 问题**（GitHub #24019、#19131）：Redis 服务重启后，客户端不会自动重连。开发期 docker-compose 重启 redis 可能踩坑。

但对我们这个项目影响不大——tokenVersion / jti 黑名单这些场景，key 过期就没了，偶尔连接丢失不会丢业务数据。生产环境才需要正经处理（比如加重连逻辑或回退 ioredis）。

## 感悟

这次和之前 TypeBox→zod 那次是**同一个错误的两种表现**：

- TypeBox 那次：照搬参考项目（elysia-admin），没核实自己栈（Elysia 原生支持 zod）
- 这次：照搬蓝图文档（stage-3），没核实运行时（Bun 内置能力）

**共性：AI 拿到"权威来源"（参考项目 / 文档）就当真理，不会主动质疑"这个来源现在还准吗"。**

以后遇到"装某个库"的建议，第一反应应该是问：
1. **Bun 是不是已经内置了？**（查 bun.sh/docs）
2. **Elysia 是不是有官方插件？**（查 elysiajs.com/plugins）
3. **文档写这个的时候是什么时候？现在过时了吗？**

这三问能挡掉大部分"无脑装包"的建议。
