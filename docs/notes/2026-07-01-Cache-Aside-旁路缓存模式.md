# Cache-Aside 旁路缓存模式

**日期**:2026-07-01
**阶段**:5.3 字典管理
**触发问题**:问"改了 DB 某条数据后马上再查，会不会查到旧缓存"。由此引出"写操作怎么处理缓存"的问题。

---

## 结论

整个项目统一用 **Cache-Aside（旁路缓存）** 模式：

```
读：查 Redis → 有则返回
            ↘ 无则查 DB → 写 Redis → 返回

写：写 DB → 删 Redis key
```

---

## 为什么写操作删 key，而不是更新缓存

删比更新简单可靠：

| 做法 | 风险 |
|------|------|
| 更新缓存 | 缓存存的数据格式可能跟 DB 不一样（比如脱敏后的列表、JSON 序列化后的对象），写错了就脏数据，而且改一次删一次能查出来（select n+1） |
| **删 key** | 删不掉最多多查一次 DB，不会读到脏数据。下次读自动重建 |

---

## 项目里所有缓存都走这个模式

| 缓存 | 什么时候删 |
|------|-----------|
| 字典项 withCache | 字典增/改/删接口里手动删 key |
| 用户权限 perms | 登录时写，角色绑定菜单/部门时在编排逻辑里删 |
| 在线用户 online | 登出/踢人时删 + TTL 自动过期 |
| token 版本号 | 只递增不删，永远不会读到旧的 |

没有例外。

---

## 补充：双删（延迟双删）

极端情况：A 写 DB 删 Redis，同时 B 读到 Redis 空，查 DB 拿到老数据写回 Redis。等 B 写完了 A 删完了，Redis 里反而剩了 B 写的老数据。

双删就是删两次：

```
A：写 DB → 删 Redis → 等 500ms → 再删一次 Redis
```

第一版不做。这个 race condition 需要"写操作的瞬间刚好有人在读"才会触发，概率极低。线上遇到了再加不迟（ponytail）。
