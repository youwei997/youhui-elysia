# treePath LIKE 通配符匹配越界

`2026-07-05` · 阶段 5 数据权限改造时踩过的坑

---

treePath 用逗号分隔整条祖先链，末尾是**自身父节点 ID**，不是自身 ID。

举个例子（`calcTreePath` 的逻辑：`父节点.treePath + "," + 父节点ID`）：
- id=1, parentId=0, treePath=`"0"`（根）
- id=2, parentId=1, treePath=`"0,1"`
- id=3, parentId=2, treePath=`"0,1,2"`
- id=4, parentId=3, treePath=`"0,1,2,3"`

查 id=3 的所有子孙（含 id=3 自身），正确写法：

```sql
-- ✅
WHERE treePath = '0,1,2'
   OR treePath LIKE '0,1,2,%'
```

`treePath = '0,1,2'` 匹配的是**父节点为 2 的节点**（即 id=3 自身）。

`LIKE '0,1,2,%'` 要求 2 后面**必须跟逗号**，所以 `"0,1,20"` 的 0 不匹配逗号，不会误命中。

---

下意识写了：

```sql
-- ❌
WHERE treePath LIKE '0,1,2%'
```

跑出来的结果看着数量对，但仔细一看——id=20 的部门也在结果里。

20 不是 3 的子部门啊。

问题是 `"0,1,20"` 确实以 `"0,1,2"` 开头。`LIKE` 做的是**纯字符串前缀匹配**，它不知道 `"2"` 和 `"20"` 是两个不同的 ID，也不知道逗号是分隔符。

更隐蔽的受害者：id=21、id=212……所有以 2 开头的 ID 都会被误拉进来，只要它们的 treePath 恰好在同一条前缀下。比如 id=212 的 treePath 是 `"0,1,20"`，它的父节点是 20，和 id=3 毫无关系。

---

后来我把这个逻辑收进了 `src/db/helpers/tree.ts` 的 `descendantsByTreePath` 函数，所有需要子树查询的地方统一走它，不让任何人再手写 LIKE 条件。

---

总结：**逗号分隔的路径做前缀匹配时，分隔符本身就是边界。漏掉逗号，LIKE 就会把同前缀的兄弟节点当成子孙吞进来。** 这不是 SQL 的 bug，是字符串匹配的语义和业务语义不一致。
