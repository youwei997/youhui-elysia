# 为什么不用 Result 容器做错误处理

`2026-06-14` · 阶段 2 初期决策

---

## 起因

一开始 AGENTS.md 里提过用 Result/match（Rust 风格的 Ok/Err 容器）做错误处理。后来看了一眼 match 的写法，就觉得不对劲，砍掉了。

## 我当时为什么不喜欢

### 第一反应：和 TS 常规代码格格不入

TS/JS 里错误处理的主流就是 `throw` + `try/catch`。突然来个：

```ts
const result = await findUser(id)
if (!result.ok) {
  return err(result.error)  // 还得包一层
}
const user = result.value
```

一眼就觉得"这不是 TS 该长的样子"。这种 match 匹配更像是**底层库级**的东西——比如你写一个解析器、一个状态机，确实需要精确区分"成功值"和"错误值"。但业务路由里用，就别扭。

### 写 match 像在写复杂逻辑判断

本来错误处理应该是**轻量**的：要么有数据，要么没数据，没了就报错。结果套一层 Result 容器，每一步都得判断 `result.ok`，写得像在做条件分支组合拳。代码量上去了，可读性还下来了。

Rust 里 Result 优雅，是因为它有模式匹配的语言级支持 + 强类型推导，match 写起来干净。TS 没这套，硬搬过来就是负担。

### 我对错误处理的预期

我印象里的错误处理应该是**简单**的，分两种情况：
1. **框架直接拦截**：像 Elysia 的 `onError`，全局兜底，业务代码不用管
2. **业务/路由层直接抛**：`throw new BizError("用户不存在", 404)`，一句话搞定

Result 容器把这两件事搅在一起了——既不是框架级拦截（因为你要手动 match），也不够轻量（因为每层都得判断 ok）。卡在中间，两头不讨好。

## 最后怎么定的

**queries 直接返回 data 或 undefined，HTTP 错误判断放 routes 层 throw**：

```ts
// queries.ts —— 纯数据操作，不知道 HTTP 是什么
export const findUserById = async (id) => {
  const rows = await db.select()...
  return rows[0]  // 没有就 undefined，让 routes 决定怎么办
}

// routes.ts —— 决定 HTTP 行为
const user = await findUserById(id)
if (!user) throw new BizError("B1001", "用户不存在", 404)
```

简单，符合主流，分层清晰。queries 是纯函数不该碰 HTTP，routes 专注编排。

## 一个小感悟

**语言/生态的好东西，不能光看它在原产地多优雅，得看搬过来顺不顺手。** Result 在 Rust 是一等公民，在 TS 是外来客。它的优雅依赖 Rust 的模式匹配和类型系统，离开那个土壤就只剩啰嗦。

以后看到"某某语言/库用了 XX 模式很优雅"，先问一句：**这个优雅是 XX 本身的，还是它依赖的那个生态给的？** 搬过来之后那个生态还在不在？
