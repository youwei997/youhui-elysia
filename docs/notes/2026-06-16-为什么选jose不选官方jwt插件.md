# 为什么选 jose 而不选 @elysia/jwt

## 背景

JWT 签名/验证需要一个库。有两个候选：官方 `@elysia/jwt` 插件和直接基于 `jose`。

## 对比

| 维度 | `@elysia/jwt` | `jose` |
|------|--------------|--------|
| 提供的能力 | `ctx.jwt.sign(payload)` / `ctx.jwt.verify(token)` | `SignJWT` / `jwtVerify` |
| 形态 | Elysia plugin，挂载后通过 ctx 调用 | 纯函数，任意模块可 import |
| 处理三层失效 | ❌ 只做验签 + exp，不碰 Redis | ❌ 同上（但我们是纯函数，搭配 Redis 查询自然） |
| 能否被 `modules/` import | ❌ 不能（plugin → modules 是反向依赖） | ✅ lib 层纯函数 |
| 额外发挥空间 | 局限于 Elysia 生命周期 | jose 支持更多算法、JWE、JWK 等 |

## 决策

选 `jose`。

关键理由：三层失效设计（tokenVersion + jti 黑名单）需要查询 Redis，@elysia/jwt 不处理这个。如果用 @elysia/jwt，JWT 的生成和验证会被拆成两半——官方插件负责验签，自己的代码负责查 Redis，逻辑割裂。而且 @elysia/jwt 是 plugin 形态，只能活在 `plugins/` 层，无法被 `modules/auth/queries.ts` 引用（反向依赖违规）。

用 jose 后，`lib/jwt.ts` 是纯函数，verifyToken 里验签 + 查 Redis 一气呵成。

## 代价

安装 jose 需要额外一个依赖（@elysia/jwt 本身也是 jose 的上层封装，但多了一个中间层）。
