# 操作日志物理删除策略

**日期**:2026-06-29
**阶段**:5.1 操作日志
**关联文件**:`src/db/schema/system/oper-log.ts`

## 背景

`sys_oper_log` 是"事件型"数据,跟业务数据(用户 / 角色 / 菜单)性质不同:

- **业务数据**:核心资产,误删代价高,需要软删兜底
- **事件数据**:流水记录,只关心"谁在什么时候做了什么",无主从关系,误删可接受

如果操作日志走软删,会产生两个问题:

1. 列表 / 详情查询永远要 `WHERE deleted_at IS NULL`,噪音
2. 5.5 定时清理任务想"删 30 天前的"时还得 `UPDATE deleted_at = NOW()`,反而更慢,不如直接 `DELETE`

## 决策

`sys_oper_log` 走 **物理删除**,不进入软删体系。

具体行为:

- **列表 / 详情查询**:`SELECT ... FROM sys_oper_log`,**不加** `WHERE deleted_at IS NULL`
- **DELETE 单条接口**:`DELETE FROM sys_oper_log WHERE id = ?`(硬删,不走 `update deleted_at = NOW()`)
- **批量清理接口**:`DELETE FROM sys_oper_log WHERE created_at < ?`(硬删)
- **5.5 定时清理任务**:`DELETE FROM sys_oper_log WHERE created_at < NOW() - INTERVAL '30 days'`(硬删)
- **不写 `deleteTime` 字段**:`auditColumns` 里的 `deleteTime` 在本表里 **不存在**,schema 中只局部复用了 `auditColumns.createTime` 这一个字段(完整复用与局部复用的区别见 `2026-06-29-auditColumns-局部复用案例.md`)

## 误删保护

- 依赖 PostgreSQL 定期 `pg_dump` 备份(运维层面保证,不在本表 schema 范围)
- 本表不进入软删,因此 **前端没有"回收站"功能**
- 如果用户强烈需要"误删恢复",后续可走"备份到冷存储"路线(见下方"未来扩展")

## 未来扩展(暂不实现)

如果将来需要"误删恢复"功能,有两种路线:

1. **加回收站接口**:复制 `sys_oper_log` 表结构到 `sys_oper_log_recycle`,DROP 前先备份,保留 7 天
2. **备份到冷存储**:删除前 dump 成 JSON / Parquet 存 S3,只保留 30 天

这两种都不在当前阶段范围,等真实业务诉求再设计。

## 适用范围

本策略 **仅适用于"事件型"数据**:

- `sys_oper_log`(操作日志,5.1)
- `sys_login_log`(登录日志,5.2)
- `sys_file`(文件元数据,5.4)—— 视业务定:如果元数据可改就走软删,不可改就物理删
- `sys_job`(定时任务配置,5.5)—— 任务有生命周期,过期即删

**业务表(用户 / 角色 / 菜单 / 部门)继续走软删策略**(`auditColumns` 五件套全部展开使用)。

## 跟"前端响应约定"的关系

AGENTS.md §4.6 规定:

- `deleteTime` / `treePath` **不输出**给前端

本表不写 `deleteTime` 字段,正好天然满足"不输出"——前端 schema 类型推导时该字段就不存在,不存在"忘了过滤"的风险。
