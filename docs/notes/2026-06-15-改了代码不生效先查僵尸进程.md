# 改了代码不生效——先查端口有没有僵尸进程

`2026-06-15` · 阶段 3 · error-handler 调试

---

## 现象

写了 error-handler plugin，装配到 app.ts，但 curl 一直返回旧的错误格式（500 + 纯文本"用户不存在"）。

反复折腾：
- 怀疑 plugin 作用域 → 改成纯函数 + 内联 onError
- 怀疑装配顺序 → 把 onError 移到路由之后
- 怀疑类型不匹配 → 调参数类型

**全没用。** 怎么改代码，curl 结果纹丝不动。

## 真相

最后 `netstat -ano | findstr :3000` 一查，**3000 端口有 3 个 bun 进程在监听**，都是之前多次 `bun dev` 留下的僵尸（Ctrl+C 没杀干净，或终端关了进程没退）。

我 curl 打的一直是某个旧进程，跑的是旧代码。新代码根本没被执行。改再多也没用。

把端口换成 8000（避开僵尸），新代码立刻生效，error-handler 返回正确的 `{code,msg,data}` 格式。

## 为什么会这样

bun 进程（bun.exe）在 Windows 上有时不会被 Ctrl+C 干净杀掉。多次启停后，旧的进程堆在后台，继续监听端口。新进程起的端口被占，要么起不来，要么……奇怪的是居然能多个进程同时 LISTEN 同一端口（Windows 的 SO_REUSEADDR 行为），请求被随机分发到某个进程。

所以"改代码不生效"的本质是：**你以为在测新代码，其实在测旧代码。**

## 排查方法

改了代码不生效，按这个顺序排查，**别急着改代码**：

```bash
# 1. 看端口被谁占
netstat -ano | findstr :3000

# 2. 如果有多个 PID，都是僵尸进程
# 3. 杀掉（需要管理员权限）
Stop-Process -Id <PID> -Force
# 或任务管理器 → 详细信息 → 结束 bun.exe

# 4. 换个干净端口启动，确认新代码生效
```

PowerShell 也能查：
```powershell
Get-NetTCPConnection -LocalPort 3000 -State Listen
Get-Process -Id <PID>  # 确认是不是 bun
```

## 教训

这次最讽刺的是——**我违反了自己之前写的 notes**。

《文档和实现会脱节》那条笔记里明明写了："任何文件改动，不能凭工具返回值判断，必须 git diff 回读。怀疑前提：当结果看起来很顺，要警惕是不是某个早期步骤根本没生效。"

结果这次 error-handler 不生效，我没去怀疑"是不是代码根本没在跑"，而是一路怀疑代码结构、类型、装配顺序，改了一圈全是空忙。

**正确的第一步应该是：确认请求打到了正确的进程。** 一个 `netstat` 就能发现僵尸进程，30 秒解决的问题，被我搞成了两小时的瞎折腾。

## 更通用的教训

这个坑不只适用于端口。任何"改了不生效"的场景，先问三个问题：

1. **代码真的在跑吗？**（进程是不是旧的？有没有热重载失败？build 缓存？）
2. **请求真的打到这个代码了吗？**（端口冲突？代理？CDN 缓存？）
3. **缓存清了吗？**（浏览器缓存？node_modules？.bun 缓存？）

确认这三点没问题，再开始改代码。否则可能像我这次，方向全错。

## 关于 bun dev 习惯

以后启 bun dev 要注意：
- 每次重启前，先确认上次的进程真的退了（任务管理器看一眼 bun.exe 数量）
- 或者固定用一个端口，发现起不来就先清僵尸
- Windows 上 bun 进程容易留僵尸，比 node 更明显（可能是 Bun 还比较新的问题）

养成习惯：**启动服务前，先 `netstat` 确认端口干净。**
