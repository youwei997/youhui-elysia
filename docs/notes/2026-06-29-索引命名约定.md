# 索引命名约定

**日期**:2026-06-29
**阶段**:5.1 操作日志
**触发问题**:阶段 5 开始建事件型表,需要建 index,但项目此前 5 张业务表都没建过显式 index

## 背景

阶段 5 前,项目业务表(`sys_user` / `sys_role` / `sys_menu` / `sys_dept` / 三张关联表)的索引情况:

- `sys_user.username`:唯一索引(Drizzle `.unique()` 自动生成 `sys_user_username_key`)
- 其他表:依赖 PG 默认主键索引

`drizzle-kit generate` 出来的索引名是 PG 默认的 `<table>_<col>_index`,例如 `sys_user_username_index`。

**问题**:这个默认名比较啰嗦(已经有 `_key` 风格了,`_index` 风格又加一套),而且阶段 5 事件型表会建很多 index,需要一套可读、grep 友好的命名。

## 决策:临时约定 `idx_<table>_<col>`

```ts
// 示例(oper-log.ts)
(table) => ({
  userIdIdx: index("idx_oper_log_user_id").on(table.userId),
  createTimeIdx: index("idx_oper_log_create_time").on(table.createTime),
  moduleActionIdx: index("idx_oper_log_module_action").on(
    table.module,
    table.action,
  ),
}),
```

**规则**:

- 普通索引:`idx_<table>_<col>`(全小写,下划线分隔)
- 复合索引:`idx_<table>_<col1>_<col2>`(按字段顺序拼接,无分隔符)
- 唯一索引:`uniq_<table>_<col>`(沿用 PG 默认风格的延伸,等阶段 5 收尾统一)
- 主键:保持默认(由 `.primaryKey()` 生成 `<table>_pkey`)

## 阶段 5 收尾时统一 review

本约定是 **临时** 的,等阶段 5 全部建表完成后(5.6 做完),会做一次全局 review:

- 如果 `idx_<table>_<col>` 风格被广泛接受 → 写成正式 ADR,业务表也补显式 index
- 如果团队更倾向 PG 默认 `<table>_<col>_index` → 改回去,5.1-5.6 表也跟着改(用一次性迁移)

**当前选择理由**:`idx_` 前缀在 `psql \d+` 列表里更显眼,前缀即语义;不依赖列顺序,grep `<table> <col>` 一下就能搜到所有相关索引。

## 跟已有索引的关系

- `sys_user.username` 的唯一索引是 `.unique()` 自动生成的 `sys_user_username_key`,**不动**
- 业务表后续如果要补显式 index,沿用本约定的 `idx_` 前缀
- 阶段 5 事件型表用本约定
- 阶段 5 收尾时,如果决定改风格,**业务表不动,事件型表跟着改**(避免大范围 churn)

## 索引选型原则(参考 AGENTS.md)

AGENTS.md 原话:"不主动加索引,**实测慢查询**才加,加完写理由"。

- 索引服务于 **已知查询模式**,不是"看着可能慢就加"
- 操作日志 3 个索引的设计依据:
  - `userId`:按用户查(管理后台看"某用户最近操作")
  - `createTime`:按时间窗清理(5.5 定时任务) + 列表按时间倒序
  - `(module, action)`:按模块 + 动作查(管理后台过滤面板)
- 阶段 5 后续表的索引按"已知查询 + 已知清理"两条线走,**不预设**
